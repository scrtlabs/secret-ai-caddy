# x402 Payment Protocol for Secret AI Caddy

The x402 subsystem adds prepaid metering and payment enforcement to the Caddy reverse proxy. AI agents authenticate with HMAC-signed requests, and Caddy enforces per-agent balances using an in-memory ledger synced with a billing backend (SecretVM). Agents with insufficient balance receive a machine-readable `402 Payment Required` response with funding instructions.

x402 runs as a parallel authentication path alongside the existing API key flow. When a request carries `X-Agent-Address` / `X-Agent-Signature` / `X-Agent-Timestamp` headers, it takes the x402 path. All other requests continue through the legacy API key validation unchanged.

## How It Works

### Request Lifecycle

```
Client (with agent headers)
  │
  ▼
┌─────────────────────────────────────────────┐
│ ServeHTTP                                   │
│  ├─ Metrics endpoint? ──yes──▶ /metrics     │
│  ├─ Blocked URL? ──yes──▶ 403               │
│  ├─ IsAgentRequest? ──yes──▶ serveX402()    │
│  └─ Legacy API key path (unchanged)         │
└─────────────────────────────────────────────┘

serveX402():
  1. Read full body (io.ReadAll — untruncated, for HMAC)
  2. Verify HMAC-SHA256 signature against agent key
     ✗ → 401 Unauthorized
  3. Read body again via BodyHandler (may truncate for token counting)
  4. Detect model, count input tokens
  5. Determine output budget (max_tokens or default)
  6. QuoteEngine.Estimate() → cost in uscrt
  7. Ledger.Reserve(agent, cost)
     ✗ insufficient → lazy hydration (ForceSync) → retry
     ✗ still insufficient → 402 Payment Required
  8. Forward to upstream (next.ServeHTTP)
     ✗ upstream 5xx → Cancel reservation, no charge
  9. Count output tokens from response
 10. SettlementEngine.Settle() → charge actual, refund unused
 11. Record usage via TokenAccumulator
 12. Record metrics
```

### Signing Protocol

Agents sign requests using HMAC-SHA256. The canonical payload is:

```
METHOD + "\n" + PATH_ONLY + "\n" + BODY + "\n" + TIMESTAMP
```

Where:
- `METHOD` is the HTTP method (e.g., `POST`)
- `PATH_ONLY` is the URL path without query string (e.g., `/v1/chat/completions`)
- `BODY` is the exact request body bytes
- `TIMESTAMP` is RFC 3339 format (e.g., `2026-04-27T18:30:00Z`)

The signature is hex-encoded and sent in `X-Agent-Signature`. Requests with timestamps outside the configured skew window (default 5 minutes) are rejected.

### Reserve-Then-Settle Pattern

Every request follows a three-phase lifecycle:

1. **Reserve** — Before forwarding, Caddy estimates the cost and deducts it from the agent's available balance. If the balance is insufficient, a 402 is returned immediately (no upstream call).

2. **Forward** — The request is sent to the upstream AI service. If the upstream returns a 5xx error, the reservation is cancelled (full refund, no charge).

3. **Settle** — After the response completes, Caddy counts actual output tokens, reprices using real usage, and refunds the difference between the reserved amount and the actual cost.

This bounds cost exposure to the reservation cap while ensuring agents are never overcharged.

### 402 Payment Required Response

When an agent's balance is insufficient, Caddy returns:

```json
{
  "agent_address": "agent-123",
  "required_amount": 150,
  "currency": "uscrt",
  "payment_url": "https://portal.example.com/fund",
  "challenge_ref": "d7nqbb4tkn1iqtr8og50",
  "message": "Insufficient balance. Please add at least 150 uscrt to continue."
}
```

Headers: `Content-Type: application/json`, `X-Payment-Required: true`.

The `payment_url` points to the Portal where the agent (or its operator) can add funds. After funding, the reconciler syncs the new balance and subsequent requests succeed.

### Balance Management

- **SpendableLedger** — In-memory, thread-safe, per-agent balance store. `Balance` represents available funds (excludes reserved amounts). Reserve deducts from balance; release/commit refunds.
- **Reconciler** — Background goroutine that periodically syncs balances from the billing backend (SecretVM). On startup, it also discovers funded agents via `ListAgents`.
- **Lazy Hydration** — On the first request from an unknown agent (e.g., after a Caddy restart), `ForceSync` is called to fetch their balance before issuing a 402. This prevents false rejections after cold starts.
- **Reservation TTL** — Stale reservations (from dropped connections) are auto-released by a background cleanup goroutine.

## Configuration

All x402 directives are optional. x402 is disabled entirely unless `x402_enabled` is set to `true`.

### Caddyfile Directives

| Directive | Type | Description | Default |
|-----------|------|-------------|---------|
| `x402_enabled` | bool | Enable x402 payment enforcement | `false` |
| `x402_agent_key` | string | Shared HMAC key (hex) or `{env.VAR}` | Required if enabled |
| `x402_agent_address` | string | Caddy's own agent address for SecretVM auth | Required if enabled |
| `x402_secretvm_url` | string | SecretVM API base URL | Required if enabled |
| `x402_payment_url` | string | Portal payment URL included in 402 challenges | Required if enabled |
| `x402_timestamp_skew` | duration | Max allowed timestamp drift | `5m` |
| `x402_reconcile_interval` | duration | Ledger sync interval | `30s` |
| `x402_default_output_budget` | int | Default max output tokens for quoting | `4096` |
| `x402_pricing_file` | path | JSON file with per-model pricing | built-in defaults |
| `x402_currency` | string | Unit of account for pricing | `uscrt` |
| `x402_reservation_ttl` | duration | Max reservation lifetime before auto-release | `5m` |

### Environment Variables

| Variable | Description |
|----------|-------------|
| `X402_AGENT_KEY` | Shared HMAC secret for agent signature verification |
| `X402_AGENT_ADDRESS` | Caddy proxy's agent identity for SecretVM API calls |
| `X402_SECRETVM_URL` | SecretVM billing backend base URL |
| `X402_PAYMENT_URL` | Payment portal URL returned in 402 challenges |

### Example Caddyfile

```caddyfile
secret_reverse_proxy {
    # Legacy auth (unchanged)
    API_MASTER_KEY {env.SECRET_API_MASTER_KEY}
    secret_node {env.SECRET_NODE}
    contract_address {env.SECRET_CONTRACT}
    secret_chain_id {env.SECRET_CHAIN_ID}

    # x402 Payment Protocol
    x402_enabled true
    x402_agent_key {env.X402_AGENT_KEY}
    x402_agent_address {env.X402_AGENT_ADDRESS}
    x402_secretvm_url {env.X402_SECRETVM_URL}
    x402_payment_url {env.X402_PAYMENT_URL}
    x402_timestamp_skew 5m
    x402_reconcile_interval 30s
    x402_default_output_budget 4096
    x402_currency uscrt
    x402_reservation_ttl 5m
}
```

### Pricing File Format

If `x402_pricing_file` is set, it should be a JSON file:

```json
{
  "default": {
    "input_cost_per_1k_tokens": 10,
    "output_cost_per_1k_tokens": 30
  },
  "models": {
    "llama3.3:70b": {
      "input_cost_per_1k_tokens": 15,
      "output_cost_per_1k_tokens": 45
    }
  }
}
```

If no file is configured, built-in defaults are used (10/30 per 1k tokens).

## Error Handling

| Scenario | HTTP Status | Behavior |
|----------|-------------|----------|
| Missing x-agent-* headers | — | Falls through to legacy API key path |
| Invalid HMAC signature | 401 | Rejected immediately |
| Stale timestamp (outside skew window) | 401 | Rejected immediately |
| Unknown agent (no ledger entry) | 402 or 200 | Lazy hydration via ForceSync; 402 only if still insufficient |
| Insufficient balance | 402 | Machine-readable challenge with funding instructions |
| Upstream error (5xx) | 5xx | Reservation cancelled (no charge), error forwarded |
| Settlement failure | 200 | Fail-open: response already sent, error logged for reconciliation |
| Reconciler sync failure | — | Logged, stale ledger retained, retry on next interval |
| Reservation TTL expired | — | Auto-released by background cleanup |

## Testing x402

### Prerequisites

- Docker and Docker Compose
- `curl`, `openssl`, `jq` (for the test script)

### Running the End-to-End Tests

The test suite uses a SecretVM simulation server and a mock AI upstream, both running as Docker containers alongside the Caddy proxy.

```bash
# 1. Build the Caddy image
docker build -t secret-reverse-proxy:latest .

# 2. Start the test environment
cd test/x402
docker compose -f docker-compose-x402.yaml up -d --build

# 3. Run the test suite
bash test_x402.sh

# 4. Clean up
docker compose -f docker-compose-x402.yaml down
```

### Test Environment

The `docker-compose-x402.yaml` starts three services:

| Service | Port | Purpose |
|---------|------|---------|
| `simulator` | 9100 | SecretVM API mock (balance, add-funds, list-agents) |
| `simulator` | 9200 | AI upstream mock (chat completions) |
| `caddy` | 8080 | Caddy proxy with x402 enabled |

The simulator pre-funds `agent-funded` with 100,000 uscrt and `agent-low` with 10 uscrt. `agent-unfunded` starts at 0.

### Making Agent Requests Manually

```bash
# Set shared secret (must match x402_agent_key)
AGENT_KEY="test-x402-shared-secret"

# Build signed request
METHOD="POST"
PATH="/v1/chat/completions"
BODY='{"model":"llama3.3:70b","messages":[{"role":"user","content":"Hello"}],"max_tokens":100}'
TIMESTAMP="$(date -u +"%Y-%m-%dT%H:%M:%SZ")"
CANONICAL="${METHOD}\n${PATH}\n${BODY}\n${TIMESTAMP}"
SIGNATURE=$(echo -ne "$CANONICAL" | openssl dgst -sha256 -hmac "$AGENT_KEY" -hex | awk '{print $NF}')

# Send request
curl -X POST http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "X-Agent-Address: agent-funded" \
  -H "X-Agent-Signature: $SIGNATURE" \
  -H "X-Agent-Timestamp: $TIMESTAMP" \
  -d "$BODY"
```
